  How synthetic data is generated

  It's a seeded deterministic model, not random rows ‚ÄĒ globalSeed=42 + the
  committed spec (data/seed/seed-config.json) is the entire dataset, so no big
  CSV is committed and every run is byte-identical. Core is
  SeasonalityModel.java. For each (tenant, category, channel, day) cell:

  value = base √ó channelShare √ó trend √ó weekly √ó monthly √ó hve(date,channel) √ó
  noise
  - base / aov / onlineShare / sparse ‚ÄĒ per category (6 categories in the spec)
  - trend ‚ÄĒ 1 + trendAnnual √ó yearsElapsed (15%/yr growth)
  - weekly / monthly ‚ÄĒ seasonality factor arrays (Mon‚ÄďSun, Jan‚ÄďDec) per channel
  - hve ‚ÄĒ high-volume-event multipliers from a calendar (Black Friday, Cyber
  Monday, Prime Day, December ramp) with per-channel magnitudes
  - noise ‚ÄĒ bounded jitter (noiseBand 0.2 ‚Üí ¬Ī10%)
  - sparse gate ‚ÄĒ intermittent categories fire only ~18% of days (cold-start
  story)
  - outlier ‚ÄĒ one one-off spike (cat_home/online, 70 days ago, √ó10)
  - returns ‚ÄĒ returnRate 0.07 netted as signed negatives

  The randomness is per-cell, seeded by hash(globalSeed, tenant, category, 
  channel, epochDay) ‚ÄĒ so it's reproducible and order-independent, and the live
  make trickle (same model, same day) continues the seeded history seamlessly.
  Two load modes share the one core: make seed emits pre-summed AggregateRows
  straight into the rollup (fast backfill); make trickle emits individual
  SaleEvents through POST /events (exercises dedupe/idempotency).

  Key point for your tenant question: because the RNG keys on
  tenantId.hashCode(), adding or renaming a tenant automatically produces its 
  own distinct data with no per-tenant tuning. But all tenants currently share
  one category catalog + seasonality, so 10 tenants come out statistically 
  similar in shape (same 6 categories, similar leader order), differing only in
  noise/absolute values.

  ---
  How to run the service

  Prerequisites: Docker Desktop, JDK 21 (your JDK 26 also builds fine), Maven, a
  browser. No Node, no AWS. Dashboard runs at http://localhost:8080.

  make run is a foreground/blocking server, so seed and forecast before you
  start it (they write to the DB directly, not through the API). Clean sequence
  from the repo root:

  # 1. Start Postgres + Redis (Flyway migrations apply on first API boot)
  make up

  # 2. Bulk-load 26 tenants of seasonal history (direct DB backfill, fast)
  make seed

  # 3. Run the forecast batch ‚Üí versioned serving rows (so mode=forecast is 
  "fresh")
  make forecast

  # 4. Boot the API + dashboard (blocks this terminal; Ctrl+C to stop)
  make run

  Then open http://localhost:8080 and pick a tenant (tenant_a‚Ä¶tenant_z), mode
  (actuals/forecast), window, channel, k.

  Optional extras (each in a second terminal while make run is up):
  make trickle   # post live SaleEvents ‚Üí watch the dashboard move (needs API 
  running)
  make demo      # Newman/Postman run against the local stack
  Standalone (no running stack):
  make test      # 150 unit tests
  make eval      # regenerate the WAPE backtest report (pure-JVM, no DB)
  Tear down (the -v wipes the Postgres volume for a truly cold next start):
  make down      # = docker compose ... down -v
  Quick health check while running: curl localhost:8080/actuator/health ‚Üí
  {"status":"UP"}.

  Now the synthetic-data explanation.

  ---
  How this project generates synthetic data

  The generator (service/topsales-datagen) is a rule-based, deterministic 
  simulation ‚ÄĒ not random rows. The whole dataset is a pure function of a fixed
  seed plus a committed config (data/seed/seed-config.json), so make seed
  reproduces byte-identical data every run and nothing large needs committing.

  The core model (SeasonalityModel.java) computes one demand value per (tenant, 
  category, channel, day) cell as a product of interpretable factors:

  value = base √ó channelShare √ó trend √ó weekly √ó monthly √ó hve(date,channel) √ó
  archetype √ó noise

  - base / aov / onlineShare ‚ÄĒ per category (electronics, grocery, ‚Ä¶)
  - trend ‚ÄĒ 1 + trendAnnual √ó yearsElapsed (steady 15%/yr growth)
  - weekly / monthly ‚ÄĒ seasonality factor arrays (per-channel weekday curve,
  12-month curve)
  - hve ‚ÄĒ high-volume events as recurring seasonality, not outliers
  (HveCalendar): Black Friday, Cyber Monday, a Prime-Day-style event, the
  December ramp ‚ÄĒ with per-channel multipliers (offline peaks on BF, online on
  CM) so the forecaster learns them
  - archetype (new in this PR) ‚ÄĒ the per-tenant shaper (scale + per-category
  weights) that makes tenant_c grocery-led, tenant_e home-led, etc.
  - noise ‚ÄĒ bounded multiplicative jitter (¬Ī10%)

  Layered on top: a one-off outlier (distinct from recurring HVEs ‚Üí
  outlier-dampening story), a sparse/intermittent category (fires ~18% of days ‚Üí
  cold-start story), and signed RETURN events (negative amounts ‚Üí idempotent
  netting story).

  The trick that makes it reproducible and order-independent: the per-cell
  randomness isn't a global RNG ‚ÄĒ it's seeded by hash(globalSeed, tenant, 
  category, channel, epochDay). So cell N never depends on cell N‚ąí1, reruns
  match exactly, and the live trickle (same model, same day) is a seamless
  continuation of the seeded history.

  Two load modes share that one core:
  - make seed ‚Üí emits pre-summed AggregateRows and bulk-inserts straight into
  the rollup table (fast backfill ‚ÄĒ trusted, bypasses the consumer). Gives the
  forecaster months of history instantly.
  - make trickle ‚Üí emits individual SaleEvents through POST /events, exercising
  the real dedupe/idempotency/upsert path. Powers the "watch the dashboard move"
  demo.

  So in the taxonomy below, this project is firmly (1) rule-based / mechanistic 
  simulation ‚ÄĒ chosen because forecasting needs known seasonality/trend to fit
  against and demonstrate; a GAN or random sampler would make WAPE meaningless
  and the chart flat.

  The different ways to generate synthetic data

  Approach: Rule-based / mechanistic simulation (this project)
  How it works: Hand-written formulas encode the structure you want
  (seasonality,
    trend, events) + bounded noise
  Best for: Time-series, demos, testing a system whose signal you must control
  Cost / caveat: You only get out the patterns you put in; not "realistic"
  beyond
    your rules

  Approach: Augmentation / resampling (SMOTE, jitter, mixup, back-translation)

‚ĚĮ So what does it indicate?                                                     

‚Źļ It indicates how confident the forecast is ‚ÄĒ the range the actual value is
  likely to land in, not just the single best guess.

  Reading one bar (cat_grocery, forecast $84,582, interval $74,650‚Äď$94,515):
  - The bar top is the model's single best estimate.
  - The whisker says: "I expect the real number to fall somewhere between
  ~$74.7k and ~$94.5k." It's the model quantifying its own uncertainty.

  What it tells you in practice:
  - Short whisker ‚Üí trust the number (stable, predictable category). cat_toys
  ¬Ī$2.3k ‚ÄĒ tight.
  - Tall whisker ‚Üí treat the number as soft (noisy/sparse/volatile category).
  cat_grocery ¬Ī$10k ‚ÄĒ much less certain, even though both are labeled HIGH
  confidence.
  - Overlapping whiskers ‚Üí the ranking may not be real. If two categories'
  intervals overlap heavily, you can't be sure #3 actually outsells #4 ‚ÄĒ the gap
  is within the noise. Here cat_home ($100.6k‚Äď$116.9k) sits clearly above
  cat_toys/cat_grocery, so that ordering is solid.

  The point of showing it: a forecast presented as one number invites false
  precision. The interval keeps the demo honest ‚ÄĒ it's the visual version of
  "this is a prediction with a margin," and it's the same interval.low/high the
  API returns and the WAPE backtest is scored against.
