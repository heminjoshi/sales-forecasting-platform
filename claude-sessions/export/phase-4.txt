     Phase 4 ‚ÄĒ Forecast serving + resilience (read path, degradation, cache)

     Context

     The platform is built through Phase 3: the batch (ForecasterJob) fits
     forecasters per
     (tenant, category, channel) and writes versioned serving rows with an
     atomic active-version
     swap; the read API (TopCategoriesController ‚Üí ActualsService) serves only
     actuals, and
     mode=forecast currently returns the same actuals aggregation relabeled
     status=pending. The
     forecast read plane does not exist yet: ForecastProvider is an
     unimplemented interface, there is
     no Redis caching, and only FRESH/PENDING of the 4-state Status enum are
     ever returned.

     Phase 4 closes that gap. It wires the precomputed serving rows into the
     read path behind a cache,
     adds the full degradation chain so reads never fail closed, and surfaces
     honest freshness status
     to the dashboard. The headline acceptance: wipe the serving table ‚Üí the 
     dashboard still renders
     (degraded, flagged). Source of truth: docs/lld.md ¬ß5 (read pipeline &
     degradation), ¬ß7 (cache),
     Build-Delivery-Plan-v3 Phase 4. Out of scope: GenAI insight (Phase 5 ‚ÄĒ
     insight stays null);
     the AWS profile / DynamoDB / CDK (Phase 7); the production React SPA.

     Confirmed decisions: (1) full design-faithful cache ‚ÄĒ tenantver bump from
     the batch + jittered
     TTL + single-flight + fail-open; (2) enhance the existing ranked chart now,
     and document a
     future true time-series overlay as a later enhancement; (3) implement via
     parallel sub-agents in
     waves.
