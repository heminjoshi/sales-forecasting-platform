     The sales-forecasting-platform repo is greenfield: git is initialized,
     .gitignore and the Claude tooling (CLAUDE.md, skills) are in place, but
     there is no build tooling, no modules, no local env, and no remote. Phase 0
     of private/Build-Delivery-Plan-v2.md is the gate everything else depends
     on — it produces a repo skeleton where every later phase just fills in
     modules that already compile and run. Goal of this plan: scaffold the Maven
     multi-module project, local Docker stack, Makefile, CI, and a private
     GitHub repo so the "walking skeleton" (Phase 2) can start immediately.

     This is the execution of the already-drafted
     claude-sessions/implementation-plan/phase-0-foundations.md, updated with
     locked decisions.

     Locked decisions

     - Build: Maven multi-module. Base package com.topsales. Modules:
     topsales-common, topsales-ingestion, topsales-forecast, topsales-insight,
     topsales-api.
     - Spring Boot 4.1.x (current GA, supported to 2027). Pin the exact latest
     4.1.x patch from Maven Central at execution (known GA: 4.1.0).
     - Java target 21 via <java.version>21</java.version>. Local JDK is 26
     (Temurin) — Boot 4.1 supports through Java 26, and --release 21 blocks
     post-21 APIs. CI pins Temurin 21 for a reproducible target.
     - GitHub: create private repo heminjoshi/sales-forecasting-platform, push
     now (gh is authed as heminjoshi, ssh). Flip to public in Phase 10 after
     history tidy + secret scan.
     - CI: GitHub Actions. ci.yml builds/tests the Maven reactor; infra.yml is a
     green placeholder until the CDK lands (Phase 7).
     - Minimal bootable API: topsales-api gets a real @SpringBootApplication +
     spring-boot-starter-web so make run starts an (empty) server. Domain model,
     endpoints, migrations stay in Phase 2.

     Environment (verified, read-only)

     - gh: logged in as heminjoshi (ssh); repo does not exist yet.
     - Toolchain present: JDK 26, Maven 3.9.16, Docker 29 + Compose v5, Node 24.

     2. Directory scaffold

     otherwise-empty dirs:

     3. Maven multi-module skeleton (the core of Phase 0)

     service/pom.xml (parent / aggregator):
     - <packaging>pom</packaging>; parent =
     org.springframework.boot:spring-boot-starter-parent:4.1.0.
     - <properties>: <java.version>21</java.version>, UTF-8 encoding.
     - <modules>: the five topsales-*.
     - Keep <dependencyManagement> minimal now; add Testcontainers/AWS BOMs in
     the phases that need them (8 / 5–7).

     Library modules topsales-common, topsales-ingestion, topsales-forecast,
     topsales-insight:
     - Each pom.xml: <parent> → aggregator, <artifactId> set,
     <packaging>jar</packaging>, no app deps yet (ingestion/forecast/insight may
     depend on topsales-common later — not required to compile now).
     - One src/main/java/com/topsales/<module>/package-info.java per module so
     it's non-empty and compiles.
     - No spring-boot-maven-plugin here (libraries must not repackage).

     topsales-api (the only bootable module):
     - pom.xml: depends on topsales-common + spring-boot-starter-web; declares
     spring-boot-maven-plugin in <build>.
     - src/main/java/com/topsales/api/TopSalesApplication.java — minimal
     @SpringBootApplication with main.
     - src/main/resources/application.yml — app name, server.port: 8080, default
     profile local (datasource/redis config added in Phase 2).
     - src/main/resources/static/.gitkeep — placeholder for the demo dashboard
     (Phase 2).

     4. Local Docker stack — local/docker-compose.yml

     - postgres:16 — env POSTGRES_DB/USER/PASSWORD, port 5432, named volume,
     healthcheck: pg_isready. Mounts local/init.sql (empty placeholder; Flyway
     owns schema from Phase 2).
     - redis:7 — port 6379, healthcheck: redis-cli ping.
     - Comment placeholder for an optional localstack profile (Phase 7).
     - local/init.sql — empty placeholder with a header comment.

     5. Makefile (thin wrappers)

     up    : docker compose -f local/docker-compose.yml up -d
     down  : docker compose -f local/docker-compose.yml down
     run   : mvn -f service/pom.xml -pl topsales-api -am spring-boot:run
     test  : mvn -f service/pom.xml test
     seed  : @echo "seed: wired in Phase 8 (data/seed)"
     demo  : @echo "demo: Postman/newman run, wired in Phase 8"
     synth : @echo "synth: CDK added in Phase 7"   # becomes: cd infra && npm ci
     && npx cdk synth

     6. CI — .github/workflows/

     - ci.yml: trigger on push + PR to main; actions/checkout@v4;

     service/pom.xml verify.
     - infra.yml: green placeholder job (echo "CDK validation added in Phase
     7"). Swapped to real cdk synth in Phase 7. Keeps the badge green now.

     --remote=origin --push --description "<generic one-liner>".
     - Watch CI: gh run watch (or gh run list) until ci.yml + infra.yml are
     green.
     - Add the CI badge to README.md; commit + push.

     - Update claude-sessions/implementation-plan/phase-0-foundations.md to
